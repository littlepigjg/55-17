# 电子书格式解析器说明

## 概述

本系统支持多种电子书格式的元数据解析，采用工厂模式和策略模式的架构设计。所有解析器都继承自 `BaseParser` 基类，实现统一的 `parse(file_path)` 接口，返回 `BookMeta` 对象。

支持的格式：EPUB、MOBI、PDF、AZW3、FB2、TXT

---

## 架构设计

### 核心类

- **BaseParser**：所有解析器的抽象基类，定义统一接口和降级策略
- **ParserFactory**：解析器工厂，根据文件扩展名自动选择对应的解析器
- **ParserConfig**：配置管理类，支持格式启用/禁用和参数配置
- **BookMeta**：电子书元数据模型

### 工作流程

1. 扫描器发现电子书文件
2. 工厂根据扩展名选择对应的解析器
3. 解析器尝试解析元数据
4. 解析失败时自动降级（使用文件名作为书名）
5. 返回 BookMeta 对象

## 各格式解析能力说明

### 1. EPUB 格式

**解析能力：**
- ✅ 书名（title）
- ✅ 作者（author）
- ✅ 语言（language）
- ✅ 简介（description）
- ✅ 出版社（publisher）
- ✅ 出版日期（publish_date）
- ✅ 标识符（identifier）
- ✅ 封面图片（cover_image）

**解析方式：**
解析 EPUB 包中的 OPF 文件，提取 Dublin Core 元数据。

**局限性：**
- 损坏的 EPUB 文件可能无法解析
- 加密的 EPUB 文件无法解析

---

### 2. MOBI 格式

**解析能力：**
- ✅ 书名（title）
- ✅ 作者（author）
- ✅ 出版社（publisher）
- ✅ 出版日期（publish_date）
- ✅ 语言（language）
- ⚠️ 封面图片（部分支持）

**解析方式：**
解析 MOBI 头部和 EXTH 记录，提取元数据信息。

**局限性：**
- 旧版 MOBI 文件元数据可能不完整
- 加密的 MOBI 文件无法解析
- 封面提取依赖 EXTH 记录中的封面索引

---

### 3. PDF 格式

**解析能力：**
- ✅ 书名（title）
- ✅ 作者（author）
- ✅ 主题（subject）
- ✅ 关键字（keywords）
- ✅ 页数（page_count）
- ❌ 封面图片

**解析方式：**
读取 PDF 文件的文档信息字典（Document Information Dictionary）。

**局限性：**
- 很多 PDF 文件不包含元数据信息
- 扫描版 PDF 通常没有元数据
- 无法提取封面图片
- 加密的 PDF 文件无法解析

---

### 4. AZW3 格式

**解析能力：**
- ✅ 书名（title）
- ✅ 作者（author）
- ✅ 出版社（publisher）
- ✅ 语言（language）
- ✅ 出版日期（publish_date）
- ⚠️ 封面图片（部分支持）

**解析方式：**
基于 MOBI 解析架构扩展，支持 KF8 头部结构。优先解析 KF8 部分的元数据，回退到旧版 MOBI 元数据。

**局限性：**
- 加密的 AZW3 文件无法解析
- KF8 格式的部分扩展元数据可能无法提取
- 封面提取依赖 EXTH 记录

**技术细节：**
- 检测 KF8 头部标识（"BOOKMOBI" + 类型 248）
- 支持从 KF8 边界记录定位 KF8 头部
- 优先使用 KF8 元数据，兼容性更好

---

### 5. FB2 格式

**解析能力：**
- ✅ 书名（title）
- ✅ 作者（author，支持多个作者）
- ✅ 语言（language）
- ✅ 出版社（publisher）
- ✅ 出版日期（publish_date）
- ✅ ISBN
- ✅ 简介（annotation）
- ✅ 类型（genre）
- ✅ 封面图片（部分支持）
- ✅ 标签（从 genre 提取）

**解析方式：**
解析 FB2 XML 文件，从 `<description>` 元素中提取 `<title-info>` 和 `<document-info>` 中的元数据。

**局限性：**
- 损坏的 XML 文件可能解析失败
- 封面图片需要内嵌在 FB2 文件中
- 部分 FB2 文件元数据可能不完整

**命名空间：**
使用 `http://www.gribuser.ru/xml/fictionbook/2.0` 命名空间。

---

### 6. TXT 格式

**解析能力：**
- ⚠️ 书名（推测）
- ⚠️ 作者（推测）
- ⚠️ 标签（从分类号推测）
- ❌ 封面图片
- ❌ 出版社
- ❌ 出版日期
- ❌ 语言

**解析方式：**
1. **文件名推测**：从文件名中提取书名和作者信息
   - 支持格式：`书名-作者.txt`、`书名_作者.txt`、`书名 by 作者.txt`
   - 支持括号内容识别：`[分类]书名.txt`、`【分类】书名.txt`
2. **内容采样**：读取文件前 10KB 内容，从首行和元数据行推测
   - 识别 `书名：xxx`、`作者：xxx` 等显式标记
   - 首行作为候选书名
   - 检测到章节标记时停止作者提取

**编码检测：**
- UTF-8（含 BOM 检测）
- UTF-16（含 BOM 检测）
- GBK / GB2312
- Big5
- CP1252
- Latin-1（最后回退）

**性能特性：**
- 仅读取文件前 10KB（可配置），避免大文件内存占用
- 支持 GB 级 TXT 文件的快速元数据提取

**局限性：**
- 元数据提取准确度依赖文件命名规范和内容格式
- 无法可靠获取作者信息（大部分 TXT 文件没有作者标记）
- 无法获取封面、出版社、出版日期等信息
- 编码检测可能不准确，尤其是内容较少时
- 采样限制可能导致元数据位于采样范围外时无法提取

**文件名解析示例：**
- `平凡的世界-路遥.txt` → 书名：平凡的世界，作者：路遥
- `活着_余华.txt` → 书名：活着，作者：余华
- `三体 by 刘慈欣.txt` → 书名：三体，作者：刘慈欣
- `[玄幻]斗破苍穹.txt` → 书名：斗破苍穹，标签：玄幻
- `【科幻】银河帝国.txt` → 书名：银河帝国，标签：科幻

## 降级策略

所有解析器都遵循以下降级策略，确保扫描过程不会因单个文件错误而中断：

1. **正常解析**：尝试完整解析文件元数据
2. **警告记录**：解析失败时记录警告日志
3. **降级处理**：使用文件名作为书名，其他字段留空
4. **继续扫描**：返回 BookMeta 对象，继续处理下一个文件

**异常处理：**
- 文件不存在 → 降级
- 文件损坏 → 降级
- 格式不支持 → 跳过（记录日志）
- 权限不足 → 降级
- 任何未预期异常 → 降级

## 配置管理

### 配置项

| 配置项 | 类型 | 默认值 | 说明 |
|--------|------|--------|------|
| enabled_formats.epub | bool | true | 启用 EPUB 格式 |
| enabled_formats.mobi | bool | true | 启用 MOBI 格式 |
| enabled_formats.pdf | bool | true | 启用 PDF 格式 |
| enabled_formats.azw3 | bool | true | 启用 AZW3 格式 |
| enabled_formats.fb2 | bool | true | 启用 FB2 格式 |
| enabled_formats.txt | bool | true | 启用 TXT 格式 |
| txt_sample_size_kb | int | 10 | TXT 文件采样大小（KB） |

### 配置文件

配置文件路径：`~/.ebook_manager/config.json`（或用户配置目录）

**示例配置：**
```json
{
  "enabled_formats": {
    "epub": true,
    "mobi": true,
    "pdf": true,
    "azw3": true,
    "fb2": true,
    "txt": true
  },
  "txt_sample_size_kb": 10
}
```

### 禁用格式

禁用某个格式后：
- 扫描时会跳过该格式的文件
- 工厂不会注册该格式的解析器
- 日志会记录跳过的文件数量

## 扩展新格式

要添加新的电子书格式支持，请按以下步骤操作：

1. **创建解析器类**，继承 `BaseParser`
2. **实现 `_parse` 方法**，返回 `BookMeta` 对象
3. **定义 `SUPPORTED_EXTENSIONS`**，指定支持的文件扩展名
4. **在工厂中注册**，将解析器添加到 `parser_classes` 列表
5. **添加配置项**，在 `ParserConfig.DEFAULT_CONFIG` 中添加默认启用配置
6. **编写单元测试**，覆盖正常、异常和边界情况

**示例：**
```python
class NewFormatParser(BaseParser):
    SUPPORTED_EXTENSIONS = (".new",)

    def _parse(self, file_path: str) -> BookMeta:
        meta = BookMeta()
        # 解析逻辑...
        return meta
```

## 性能考虑

- **TXT 格式**：仅读取前 10KB，内存占用小，适合大文件
- **FB2 格式**：XML 解析可能占用较多内存，大文件需注意
- **EPUB 格式**：需要解压 ZIP 包，IO 开销较大
- **PDF 格式**：需要解析 PDF 结构，大文件可能较慢

## 常见问题

**Q: 为什么有些 TXT 文件无法提取作者？**
A: TXT 格式没有标准的元数据存储方式，作者信息依赖于文件名格式或内容中的显式标记。大部分 TXT 文件不包含作者信息。

**Q: 为什么 PDF 文件的书名经常是文件名？**
A: 很多 PDF 文件没有设置文档元数据，系统会自动降级使用文件名作为书名。

**Q: 如何提高 TXT 元数据提取的准确率？**
A: 建议使用规范的文件命名格式，如 `书名-作者.txt`，或在文件开头添加 `书名：xxx`、`作者：xxx` 等标记。

**Q: 加密的电子书能解析吗？**
A: 不能。加密的电子书文件（如 DRM 保护的 AZW3）无法解析元数据，系统会自动降级使用文件名作为书名。
